# detectOverflow를 "개념"으로 이해하기 — 멘탈 모델 편

> 이 글은 `detectOverflow`의 **구현이나 계산식이 아니라, 머릿속에 남길 그림**을 다룹니다.
> 코드 한 줄 몰라도 "이 함수가 *무엇을* 하고 *왜* 그렇게 설계됐는지"가 끝까지 남는 것이 목표입니다.
> 함수 시그니처·뺄셈 공식·DOM 변환 코드 같은 각주는 일부러 뺐습니다. 남기는 건 **비유와 예시, 그리고 설계 의도**뿐입니다.

---

## 0. 한 문장으로 — 이 함수는 "센서"다

flip은 뒤집고, shift는 밀고, size는 줄이고, hide는 숨깁니다. 그런데 이들이 *판단을 내리기 전에* 공통으로 묻는 질문이 하나 있습니다.

> **"지금 이 위치, 어느 방향으로 얼마나 잘렸지?"**

이 질문에 **숫자로** 답하는 게 `detectOverflow`입니다. 자신은 아무것도 고치지 않습니다. 측정만 합니다.

```
   detectOverflow (센서)  ──→  { top, right, bottom, left }
                                      │
              ┌───────────────┬───────┴───────┬──────────────┐
            flip            shift            size           hide
          (뒤집기)          (밀기)           (줄이기)        (숨기기)
```

체온계가 열을 내리지 않고 온도만 알려주듯, 이 함수는 **보정의 근거가 되는 숫자만** 만듭니다. 나머지 미들웨어가 그 숫자를 받아 행동합니다. 이 "측정과 행동의 분리"가 글 전체를 관통하는 첫 번째 멘탈 모델입니다.

---

## 1. 문제 — 툴팁이 화면 밖으로 삐져나간다

버튼(reference) 위에 툴팁(floating)을 올렸는데, 버튼이 화면 맨 위에 있으면 툴팁이 위로 잘립니다.

```
   ┌──────────────── 화면 ────────────────┐
   │ ▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒  ← 툴팁 윗부분이 잘림   │
   │        ┌──────────────┐               │
   │        │   TOOLTIP    │               │
   │        └──────────────┘               │
   │        ┌──────────────┐               │
   │        │    BUTTON    │               │
   │        └──────────────┘               │
   └────────────────────────────────────────┘
```

flip이 "그럼 버튼 *아래로* 뒤집자"고 결정하려면, 먼저 **"지금 위로 얼마나 잘렸는가"를 정확히 알아야** 합니다. 막연히 "잘린 것 같다"로는 보정할 수 없습니다. 그 "얼마나"를 숫자로 만드는 일이 모든 충돌 보정의 출발점입니다.

---

## 2. 해결의 핵심 비유 — "방"과 "가구"

해결책은 단순합니다. **사각형 두 개를 비교**합니다.

```
   ┌─────────────────────────────────────────┐
   │  방 (clipping) = 가둬두는 영역             │
   │                                          │
   │        ┌──────────────────┐              │
   │        │ 가구 (element)    │              │
   │        │  = 검사 대상      │              │
   │        └──────────────────┘              │
   └─────────────────────────────────────────┘
```

- **방** = 잘라내는 경계. 화면, 혹은 `overflow: hidden`인 스크롤 박스처럼 "자식을 가둬두는" 영역.
- **가구** = 그 안에 놓인 검사 대상. 보통은 툴팁.

방의 네 벽과 가구의 네 변 사이 거리를 재면, "각 방향으로 얼마나 삐져나갔나"가 나옵니다. 방에 가구를 들일 때 "문틀 밖으로 삐졌나?"를 재는 그 직관, 그게 전부입니다.

> 🏠 방이 또 다른 방 안에 있다면(스크롤 박스 안의 스크롤 박스), 기준은 "겹치는 가장 좁은 벽"이 됩니다. 여러 경계를 다 통과해야 보이니까요.

---

## 3. 출력의 언어 — 부호 규약이라는 "공용 프로토콜"

센서가 내는 숫자 4개는 이런 모양입니다.

```
   { top: 28, right: -50, bottom: -300, left: -50 }
       │            │
       │            └─ 음수 = 아직 여유 (안 잘림)
       └─ 양수 = 그 방향으로 28px 넘침 (잘림)
```

규약은 딱 한 줄입니다.

```
   −∞ ◄────────────── 0 ──────────────► +∞
      여유 많음       딱 맞음        많이 넘침
```

**`+` = 넘침, `−` = 여유, `0` = 경계에 딱.**

이 단순한 약속이 왜 중요한가? **부품들을 서로 모르게 만들기 때문**입니다.

flip은 detectOverflow의 내부를 전혀 몰라도 됩니다. 그저 `if (위로_넘침 > 0)` 한 줄이면 "위로 잘렸구나, 뒤집자"를 압니다. shift도, size도, hide도 마찬가지입니다. 센서가 "넘치면 +"라는 *공용 언어*로만 말하면, 소비자들은 센서의 속사정을 몰라도 각자 일합니다.

> 🔑 **멘탈 모델:** 부호 규약은 단순한 표기법이 아니라 **부품 간의 계약(프로토콜)**입니다. "어떻게 쟀는지"는 숨기고 "넘쳤는가/여유가 있는가"만 공유함으로써, 센서와 미들웨어가 서로 독립적으로 발전할 수 있게 합니다. 좋은 인터페이스가 늘 그렇듯, 적게 약속할수록 결합이 느슨해집니다.

방향마다 부호가 일관되게 나오는 비결도 직관 하나로 정리됩니다. **항상 "바깥쪽 가장자리 − 안쪽 가장자리"**. 그래야 넘치면 양수가 됩니다. 위/왼쪽은 방이 바깥이고, 아래/오른쪽은 가구가 바깥이라 빼는 방향만 갈릴 뿐, 규칙은 하나입니다.

```
   위로 삐짐:  방의 위 벽 − 가구의 위 변 = +20  "위로 20 넘침"
   아래로 삐짐: 가구의 아래 변 − 방의 아래 벽 = +20  "아래로 20 넘침"
                └ 둘 다 "바깥 − 안쪽" → 넘치면 +
```

## 4. 가장 중요한 "왜" — 두 사각형은 서로 다른 자(尺)로 살고 있다

여기가 이 함수에서 제일 깊고 제일 헷갈리는 지점입니다. 뺄셈이 의미를 가지려면 **두 사각형이 같은 좌표계**여야 합니다. "내 방 기준 cm"와 "지도 기준 km"를 그냥 빼면 무의미하듯이.

그런데 정렬은 **비대칭**입니다. 방은 그냥 두고, 가구만 변환합니다. 왜일까요?

### (1) reference ≠ 원점

먼저 두 단어를 분리해야 합니다.

- **reference** = "어디에 놓을지" 계산할 때 쓰는 **목적지** (그 버튼).
- **offsetParent** = 툴팁의 좌표가 *실제로 측정되는* **원점**.

내비게이션으로 치면 reference는 "도착지 주소", offsetParent는 "지도의 (0,0)"입니다. 완전히 다른 역할이죠.

### (2) 왜 CSS는 화면이 아니라 offsetParent를 원점으로 삼나

이게 진짜 질문입니다. "전부 그냥 화면 기준이면 편할 텐데, 왜 중간에 부모를 원점으로 끼워넣지?"

답: **자식이 부모와 함께 움직이게 하려고.**

만약 `position: absolute` 요소의 좌표가 *화면 절대 기준*이었다고 상상해 봅시다.

```
   ❌ 좌표가 화면 절대값이라면:

   카드(부모)가 스크롤로 움직임
   → 카드 안 툴팁의 좌표는 고정값이라 그 자리에 박혀 있음
   → 부모는 떠났는데 자식만 허공에 남음 💀
```

CSS는 이걸 "부모 상대 좌표"로 풀었습니다.

```
   ✅ 좌표가 부모(offsetParent) 기준이면:

   카드가 100px 이동하면
   → 툴팁의 "부모 안에서의 자리"는 그대로
   → 자식이 자동으로 100px 따라감 ✓
```

> 🔑 **멘탈 모델:** offsetParent가 존재하는 이유는 **지역 좌표계(local coordinate system)**라는 보편적 설계 패턴입니다. 게임 엔진의 부모-자식 트랜스폼, SVG의 중첩 좌표계, UI 레이아웃 — 전부 같은 원리입니다. *"각 컨테이너가 자기만의 (0,0)을 가지면, 자식은 전역 위치를 몰라도 되고, 컨테이너가 움직여도 자식이 따라온다."* 복잡한 세상을 "각자 자기 동네 좌표만 알면 되는" 작은 문제로 쪼개는 분할정복입니다.

### (3) 그래서 좌표계가 둘로 갈린다

```
   화면 전역 좌표계 (0,0 = 화면 왼쪽 위)
   ┌──────────────────────────────────────────┐
   │  (50,80)                                  │
   │   ●━━━━━━━━━━━━━━━━━━━┓ offsetParent        │
   │   ┃ 지역 좌표계         ┃ (자기만의 0,0 = ●)  │
   │   ┃   ┌──────────┐    ┃                    │
   │   ┃   │ FLOATING │ ●로부터 (10,20)          ┃ │
   │   ┃   └──────────┘    ┃                    │
   │   ┗━━━━━━━━━━━━━━━━━━━┛                    │
   └──────────────────────────────────────────┘

   화면상 진짜 위치 = 부모(50,80) + 지역(10,20) = (60,100)
```

위치 계산이 내놓는 툴팁의 좌표는 **지역 좌표** `(10,20)`이지, 화면 절대 좌표 `(60,100)`이 아닙니다.

### (4) 그래서 변환이 필요하다 (그리고 한쪽만 한다)

- **방(clipping)**은 애초에 화면 전역 좌표로 *만들어져* 옵니다. 경계가 무엇이든 본질적으로 "화면 어디에 있나"로 측정되기 때문입니다.
- **가구(floating)**는 "부모와 함께 움직이려고" 지역 좌표로 *살고 있습니다.*

원점이 다른 둘을 그냥 빼면 거짓말이 나옵니다.

```
   ❌ 그냥 빼면:  방.top(0) − 가구.top(20) = −20  "여유 있음" ← 틀림!
   ✅ 가구를 화면 좌표로 끌어올린 뒤:  0 − 100 = −100 ← 맞음
```

> 🔑 **멘탈 모델:** 변환이 비대칭인 건 게으름이 아닙니다. **두 값이 태생적으로 다른 좌표계에 살기 때문**입니다. 방은 처음부터 전역에 살고, 가구는 "부모를 따라다니려고" 지역에 삽니다. 비교 직전, 가구만 전역으로 끌어올려 같은 자에 맞춥니다. 이건 "한쪽을 봐주는" 게 아니라, **비교의 전제 조건을 함수가 책임지는** 것입니다.

(`position: fixed`는 요소가 처음부터 화면을 기준으로 살기 때문에, 이 변환이 거의 "하나 마나"가 됩니다. "부모를 따라가느냐(absolute) vs 화면에 고정이냐(fixed)"의 차이가 곧 "지역 좌표냐 전역 좌표냐"의 차이입니다.)

## 5. scale과 padding — 측정의 두 가지 "각도 조정"

두 개념만 비유로 짚고 넘어갑니다. 둘 다 계산이 아니라 *관점*입니다.

### scale — "줄이는 게 아니라 단위 환산"

부모에 `transform: scale(2)`가 걸리면, "화면에 그려진 픽셀"과 "좌표가 쓰는 논리 픽셀"이 어긋납니다. 센서가 잰 "넘침"은 화면 픽셀인데, 그 숫자를 받아 미는 shift는 논리 픽셀을 다룹니다.

```
   지도에서 실제 거리 2km를 축척으로 나눠 "지도상 4cm"로 바꾸는 것과 같다.
   → 실제 거리를 줄인 게 아니라, 같은 거리를 다른 단위로 말한 것뿐.
```

> 🔑 scale 보정은 측정을 **약하게 만드는 게 아니라**, "화면에서 잰 넘침"을 "좌표가 알아듣는 단위"로 **번역**하는 일입니다. 번역을 안 하면 shift가 두 배로 밀어 과보정합니다.

### padding — "가상의 판정선"

"경계에 딱 닿기 전, 이만큼 못 미쳐도 넘친 걸로 쳐줘"라는 여백입니다. 판정선을 안쪽으로 당겨, flip/shift가 *조금 더 일찍* 작동하게 합니다.

```
   판정선을 벽보다 8px 안쪽에 그어두면
   → 툴팁이 화면 가장자리에 딱 붙기 전에 미리 보정
   → "아슬아슬하게 붙은" 어색함 방지
```

둘 다 "무엇을 잘못 쟀나"가 아니라 "어떤 기준으로 볼까"를 조정하는 손잡이입니다.

## 6. 두 번째 큰 "왜" — portal과 교차검사의 트레이드오프

offsetParent만큼 중요한 두 번째 설계 서사입니다. 단계적으로 갑니다.

### (1) 왜 툴팁을 멀리 떨어진 곳에 렌더링하나 (portal)

툴팁을 버튼의 DOM 자식으로 두면, 버튼의 조상 중 `overflow: hidden`이 있을 때 **그 조상이 툴팁을 잘라버립니다.**

```
   ❌ 툴팁을 버튼 옆(자식)에 두면:

   ┌──── 카드 (overflow: hidden) ────┐
   │  [BUTTON]                       │
   │     ┌──────────────┐            │
   │     │  TOOLTIP    ▒▒▒▒▒▒▒▒▒▒▒▒▒ │ ← 카드 밖이라 잘림! 💀
   │     └──────────────┘▒▒▒▒▒▒▒▒▒▒│
   └──────────────────────────────────┘
```

그래서 툴팁을 **`document.body`의 직속 자식으로 물리적으로 옮겨** 렌더링합니다. 이게 portal입니다.

```
   ✅ portal로 body에 두면:

   ┌──── 카드 (overflow: hidden) ────┐
   │  [BUTTON]                       │
   └──────────────────────────────────┘
       ┌──────────────┐
       │   TOOLTIP    │  ← body 직속이라 안 잘림 ✓
       └──────────────┘    (위치는 좌표로 버튼 옆에 맞춤)
```

> 🔑 portal의 존재 이유: *"잘라내거나 가두는 조상의 영향권 **밖**으로 물리적 위치를 빼내, 어디서든 온전히 보이게 한다."*

### (2) 그런데 portal이 새 문제를 만든다

물리적 위치를 옮긴 대가로, **버튼과 툴팁이 서로 다른 "방"에 살게 됩니다.**

```
   ┌── 버튼의 방 (카드) ──┐
   │  [BUTTON]           │
   └─────────────────────┘
       ┌─────────────┐  ← 툴팁은 body의 방 (전혀 다른 방!)
       │  TOOLTIP    │
       └─────────────┘
```

이제 "툴팁이 화면을 넘치나?"만 봐서는 **답할 수 없는 질문**이 생깁니다. 예를 들어, 사용자가 카드를 스크롤해서 버튼이 사라졌는데, 툴팁은 body에 있어서 멀쩡히 보이는 상황.

```
   ┌── 카드의 방 ──┐
   │ ░░░░░░░░░░░░ │ ← 버튼이 스크롤로 사라짐 (안 보임)
   └───────────────┘
       ┌─────────────┐
       │  TOOLTIP    │  ← ❌ 가리킬 버튼 없이 혼자 떠 있음 (유령 툴팁)
       └─────────────┘
```

일반 검사는 "툴팁이 화면 안에 있나?"만 봅니다. 툴팁은 멀쩡히 화면 안이니 **"문제 없음"**이라 답합니다. 버튼이 사라진 사실을 영영 모릅니다.

### (3) 그래서 "교차검사"가 필요하다

해법은 **한 요소를 *다른* 요소의 방 기준으로 재는** 것입니다.

| 검사 | 무엇을(가구) | 어느 방에서 | 묻는 것 |
|---|---|---|---|
| 일반 | 툴팁 | 툴팁의 방 | "화면을 넘치나?" → 위치 보정 |
| referenceHidden | 버튼 | 버튼의 방 | "버튼이 사라졌나?" → 숨김 |
| escaped | 툴팁 | **버튼의 방** | "툴팁이 버튼 영역을 벗어났나?" → 숨김 |

버튼을 버튼의 방 기준으로 재보면 "버튼이 방 밖으로 완전히 사라졌다"가 잡힙니다. 그럼 의미 없어진 툴팁을 숨길 수 있습니다.

> ⚠️ 중요한 구분: 교차검사는 **위치를 고치는 게 아니라 "관계가 깨졌는지"를 감지**합니다. 숨김 여부는 검사 결과를 받은 쪽이 결정합니다. 측정과 행동의 분리(0절)가 여기서도 반복됩니다.

### (4) 일반화 — "떼어놓은 대가"

툴팁을 "온전히 보이게" 하려고 가두는 조상 밖으로 빼내는 순간은 `overflow: hidden`만이 아닙니다. z-index 쌓임 맥락(`transform`/`opacity`/`filter`), `<dialog>`·모달 같은 별도 레이어 — **portal을 부르는 모든 상황**이 같은 트레이드오프를 만듭니다.

> 🔑 **멘탈 모델:** *"보이게 하려고 떼어놓으면, 둘의 관계가 아직 유효한지는 따로 확인해야 한다."* 교차검사는 바로 이 트레이드오프의 해법입니다. 한쪽 방만 봐서는 알 수 없는 "공간적 관계의 단절"을, 상대의 방 기준으로 재서 알아냅니다.

## 7. 전체를 관통하는 설계 철학

조각들을 다 모으면, 하나의 일관된 태도가 보입니다.

```
   "측정의 더러운 일은 환경(platform)에 위임하고,
    자신은 좌표 정렬과 부호 있는 산수만 한다."
```

| 설계 결정 | 무엇을 위해 |
|---|---|
| 측정만 하고 고치지 않음 | 센서와 행위자의 분리 → 미들웨어 독립성 |
| 부호 규약(+넘침/−여유) | 공용 프로토콜 → 부품 간 결합 최소화 |
| 두 사각형을 같은 좌표계로 정렬 | 비교의 전제를 함수가 책임 |
| 환경 측정은 외부에 위임 | 순수한 핵심 → DOM·React Native·Canvas 어디서든 동작 |
| "무엇을(가구) × 어디서(방)"를 독립 축으로 | 일반 검사와 교차검사를 같은 함수로 표현 |

마지막 한 줄로 압축하면 이렇습니다.

> `detectOverflow`는 **"방"과 "가구"를 같은 좌표계로 정렬한 뒤, 네 변의 거리를 재서 "넘친 양(+)/여유(−)"를 돌려주는 순수한 센서**다. 측정의 복잡함은 환경 뒤로 숨기고, 자신은 부호 있는 비교만 한다. 그리고 그 단순한 출력이, 위를 뒤집고 옆을 밀고 크기를 줄이고 숨기는 모든 보정의 공통 언어가 된다.
